# Video Generation ArXiv Daily Site Design

## 1. 项目目标

构建一个先在本机端口运行、后续可迁移到公网的网站。网站每天自动从 arXiv 获取 `cs.CV`、`cs.AI`、`cs.LG` 分类中“前日访问量最高”且与 video generation 相关的论文，强制下载并解析 PDF 正文，自动生成中文整理内容，并按日期归档后展示在网站上。

第一版目标是尽快跑通完整链路：

- 每天一次自动抓取前日访问量榜单并发布
- 自动生成整理结果并直接发布
- 网站提供首页、归档页、详情页
- 内容按日期持久化保存
- 记录榜单统计日期与访问热度排名
- 当前只支持本地运行，后续可升级为公网部署和数据库驱动

## 2. 范围与非目标

### 范围

- 抓取 arXiv 中 `cs.CV`、`cs.AI`、`cs.LG` 分类的前日高访问论文
- 使用关键词过滤：
  - `video generation`
  - `text-to-video`
  - `image-to-video`
  - `video diffusion`
- 强制下载并解析 PDF 正文
- 基于正文调用大模型生成结构化中文内容
- 自动发布到本地网站
- 以日期为单位存储和归档内容

### 非目标

- 第一版不做管理后台
- 第一版不做人工审核流
- 第一版不做搜索功能
- 第一版不做标签分类页
- 第一版不接数据库

## 3. 用户体验与页面结构

第一版网站包含 3 类页面：

### 首页

展示最近发布的论文整理列表。每条列表项至少展示：

- 论文标题
- arXiv 发布时间
- 本站归档日期
- 榜单统计日期
- 访问热度排名
- 一句话结论
- 原文链接

### 归档页

按日期查看当天已发布的论文整理结果。用户可以浏览某一天发布了哪些论文。

### 详情页

单篇论文页面完整展示以下字段：

- 论文标题
- arXiv id
- 作者
- 原文链接
- PDF 链接
- arXiv 发布时间
- 本站归档日期
- 榜单统计日期
- 访问热度排名
- 中文摘要
- 一句话结论
- 方法亮点
- 潜在应用
- 局限性

## 4. 系统设计

系统拆分为两个部分：

### 4.1 内容生成管线

每天运行一次，负责：

1. 查询 arXiv
2. 获取 `cs.CV`、`cs.AI`、`cs.LG` 分类的前日访问量榜单
3. 基于关键词过滤论文
4. 下载 PDF
5. 解析 PDF 正文
6. 调用大模型生成中文整理内容
7. 校验输出结构
8. 写入归档内容
9. 更新首页和归档索引

### 4.2 静态优先展示站

网站本身不直接访问 arXiv，不临时读取 PDF，不在请求时调用模型。网站只读取本地已生成的归档内容并渲染页面。

该设计满足以下目标：

- 降低网站运行复杂度
- 避免抓取失败影响网站访问
- 为后续升级数据库方案保留空间

## 5. 技术选型

第一版推荐使用 Python 单栈实现：

- Web 服务：`FastAPI`
- 模板渲染：`Jinja2`
- 内容采集与发布管线：Python 脚本
- PDF 解析：`PyMuPDF`
- 模型接口：可切换的 `LLM Adapter`
- 本地调度：先提供手动命令，后接 Windows 定时任务
- 存储：本地 `JSON` 文件

选择原因：

- PDF 下载、解析、文本处理适合 Python
- 网站结构简单，服务端模板足够满足第一版
- 模型适配层可保留后续切换到本地模型的能力

## 6. 内容存储设计

按日期归档，目录结构建议如下：

```text
content/
  2026-05-10/
    index.json
    <slug>.json
    <slug>.json
  2026-05-11/
    index.json
    <slug>.json
```

说明：

- `index.json` 保存当天列表页所需字段
- 单篇 `<slug>.json` 保存完整整理内容
- `<slug>` 可由 `arxiv-id` 或标题安全转换生成

每篇论文的数据结构至少包含：

- `title`
- `arxiv_id`
- `authors`
- `source_url`
- `pdf_url`
- `published_at`
- `archived_at`
- `traffic_date`
- `view_rank`
- `view_count`（如果数据源可提供）
- `summary_zh`
- `one_sentence_takeaway`
- `method_highlights`
- `applications`
- `limitations`

## 7. 每日数据流

每日任务按以下顺序执行：

1. 从 arXiv 可公开访问的榜单页或统计源中，获取 `cs.CV`、`cs.AI`、`cs.LG` 分类的前日访问量榜单
2. 读取榜单中的访问热度排序信息
3. 对榜单内论文的标题和摘要执行关键词过滤
4. 对命中的论文按访问热度排序逐篇下载 PDF
5. 解析 PDF 正文文本
6. 将正文及元数据发送到模型接口
7. 生成结构化中文整理结果
8. 校验字段完整性与格式
9. 保存为当天归档内容
10. 生成或刷新首页最新列表与归档索引

为了控制第一版运行时间和生成成本，默认仅发布“前日访问量榜单中，过滤后排名最高的前 5 篇论文”，该数量需配置化。

## 8. 质量控制

内容发布前需要经过 3 层校验：

### 8.1 输入校验

- 必须属于指定 arXiv 分类
- 必须出现在前日访问量榜单中
- 必须命中关键词规则
- 必须成功下载 PDF
- 必须成功解析 PDF 正文

### 8.2 输出校验

模型输出必须包含所有必填字段：

- 中文摘要
- 一句话结论
- 方法亮点
- 潜在应用
- 局限性

若缺失字段、为空或结构不符，则判定该篇失败。

### 8.3 发布校验

仅通过前两层校验的结果才能写入归档并显示在网站中。

## 9. 失败处理策略

### 单篇论文失败

- PDF 下载失败：跳过，不发布，记录失败原因
- PDF 解析失败：跳过，不发布，记录失败原因
- 模型生成失败：自动重试 1 次，仍失败则跳过
- 模型输出字段不完整：自动重试 1 次，仍失败则跳过

### 批量任务失败

- 若当天整体任务中断，不覆盖既有网站内容
- 失败信息写入运行日志，供后续排查
- 若前日访问量榜单源不可用，不发布新内容并保留现有网站内容

### 去重策略

- 以 `arxiv_id` 为主键去重
- 已归档论文不重复发布

## 10. 运行模式

第一版支持以下运行方式：

- 本地启动 Web 服务，占用指定端口访问网站
- 本地执行每日采集发布命令
- 后续用 Windows 定时任务每天触发采集命令

当前运行边界：

- 主要服务于本机访问
- 部署结构需为未来迁移到公网保留空间

## 11. 后续升级路径

第二阶段可以在保留内容生成管线的前提下升级：

- 将 `JSON` 归档替换为数据库
- 增加搜索能力
- 增加标签页或专题页
- 增加后台审核和人工修订能力
- 将模型适配层切换为本地模型或多模型策略
- 部署到公网服务器

## 12. 测试策略

第一版至少覆盖以下验证：

- 前日访问量榜单抓取与解析测试
- 关键词过滤规则测试
- 归档文件生成测试
- 模型输出结构校验测试
- 首页渲染测试
- 归档页渲染测试
- 详情页渲染测试
- 本地启动后的手工冒烟检查

## 13. 设计结论

第一版采用“内容生成管线 + 静态优先展示站”的方案，以 Python 单栈落地。该方案最适合当前目标：

- 先在本机端口快速运行
- 每天一次自动抓取前日高访问论文、整理、归档、发布
- 确保只发布成功解析 PDF 正文并完成结构化总结的论文
- 为后续迁移到公网和数据库方案保留清晰升级路径
